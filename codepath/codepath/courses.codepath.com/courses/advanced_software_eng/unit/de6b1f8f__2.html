Slides

Communication

Communication is a key part of the interview. As you're working through problems, you should be constantly communicating with the interviewer. You should treat the interview as a collaborative process of coming up with a solution for the problem. Be respectful of the interviewer and actively listen to what they're saying. You're not just being evaluated for coming up with a solution, but how you behave throughout the interview.

  1. Understanding the problem- When you're first introduced to the question, don't rush into coding the solution. Be sure that you understand what the problem is asking for by coming up with example input/ outputs and verifying with the interviewer that the input/ output are what they expect. Don't be afraid to ask questions to clarify the problem!

  2. Working through the solution- After you've understood the problem, engage in a dialogue with the interviewer about the approach you've come up before coding up the solution. Walk them through your thought process and they could possibly help guide you if you're going in the wrong direction.

  3. Coding- As you're implementing the solution, be sure to keep communicating. With every few lines of code, you should be explaining what you're aiming to accomplish with the code.

Stuck? Keep talking! If you're stuck during an interview, tell the interviewer what solution you've come up with so far, even if it is an inefficient brute force solution. Talk about all the approaches you've thought of. The interviewer generally wants to help you, but it's hard to help someone who's not talking. They can usually give you a good hint if they know exactly what you're thinking.

In class exercise instructions

Directions

  1. Introductions - Each person should introduce themselves to the other in the group including name, where you're from.

  2. Create a new codeinterview session document to work on.

  3. Talk through problems - Select one person to lead the discussion on each question and rotate for each question. The leader will be coding up the solution, but the group should work collaboratively on coming up with the solution together. This isn't just about getting a correct solution but also effective communication.

    • Come up with a few new example input/ outputs
    • Talk about the edge cases
    • Talk about different approaches
    • Pseudocode
    • Code the solution (talk throughout this segment!)
    • Test your solution

  4. Review solution - After completing a problem, scroll down to the bottom of this tab to find the solution links and discuss/compare your solution with the solution given. Is the one you've developed less efficient or less concise? Repeat this and work through the exercises below.

As you go through the questions below, be sure to rotate the person who is leading each question.

Challenge 1 - Implement a list representation of a minimum heap

Write a heap class that represents a minimum heap using an array. Implement the insert method for this min heap class.

min_heap = MinHeap()
min_heap.insert(2)
min_heap.insert(4)
min_heap.insert(1)
# Underlying array should look like: [1, 4, 2]

If time permits, implement the delete_min() method.

In Class Exercises Set #2

Challenge 1 - Hash table word count (10 - 15 minutes)

Write a program which takes as its input a String of natural language text and outputs a HashMap<String,Integer> whose keys are the unique words in the input and whose values are the number of times those words occur. The algorithm should be case-insensitive (e.g. "Program" and "program" would count as the same word) and ignore punctuation and whitespace.

Example: Given the input "To be or not to be, that is the question", the outputted HashMap would contain 8 entries, with two words having a count of 2 and six words having a count of 1:

"to"        → 2
"be"        → 2
"or"        → 1
"not"       → 1
"that"      → 1
"is"        → 1
"the"       → 1
"question"  → 1

Solution for Challenge #1

Challenge 2 - Multimaps (10 - 15 minutes)

One common pattern when using hash tables requires building a Map whose values are Collection instances. In this challenge, we'll take the output of the previous challenge and invert it.

Write a program that takes as its input a HashMap<String,Integer> and returns a HashMap<Integer,HashSet<String>> containing the same data as the input map, only inverted, such that the input map's values are the output map's keys and the input map's keys are the output map's values.

Example: Consider the example output for Challenge #1. Using that map as the input, the output for this challenge would be:

2 → ["to", "be"]
1 → ["or", "not", "that", "is", "the", "question"]

Solution for Challenge #2

Challenge 3 - Design a minimum stack (15 - 20 minutes)

Design a stack that supports push, pop, top, and retrieving the minimum element in constant time. These are the operations you should implement for this data structure.

push(x) -- Push element x onto stack.
pop() -- Removes the element on top of the stack.
top() -- Get the top element.
get_min() -- Retrieve the minimum element in the stack.

Example input:

min_stack = new MinStack()
min_stack.push(-2)
min_stack.push(0)
min_stack.push(-3)
min_stack.get_min()   # Returns -3.
min_stack.pop()
min_stack.top()       # Returns 0.
min_stack.get_min()   # Returns -2.

Instructions for in class mock interviews

Refer to this mock interview guide

Introductions

Introduce yourselves to each other! What school do you go to? Where are you from?

Mock interview prep (10 minutes)

Designate one person as person A, the other as person B.

Person A will review this problem and then ask person B this problem

question

Person B will review this problem and ask person A this question

question

Mock interviewing (40 minutes each round)

There will be 2 rounds of mock interviewing. Each round will be 40 minutes long and there should be a feedback period following each interview.

  1. Person A will begin the first round of interviews as the interviewer

    • Interviewer introduces the problem, interviewees clarify the question
    • Interviewees come up with example input/ output and write down pseudocode
    • Actual coding
    • End with a quick time/ space complexity analysis even if the coding is not complete.

  2. After the interview is over, provide feedback for how the session went.

    • Example feedback for the interviewees:

      • What did the interviewee do well?
      • Did the interviewees come up with an example input/ output before planning out the approach?
      • Did the interviewees plan sufficiently before coding?
      • Were the interviewees communicative throughout?

    • Example feedback the interviewer:

      • What did the interviewer do well?
      • Did the interviewer clearly state the question?
      • Did the interviewer help along the way?

  3. Repeat steps 1 & 2 with person B as the interviewer.
