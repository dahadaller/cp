Week 11 - Review

This week will be a project-centered review week. We will review key topics such as Greedy Algorithms, Dynamic Programming, and Backtracking while working on a project that might involve successfully using one or more of these techniques

Goals

  • Review advanced algorithms: Greedy, Dynamic Programming, Backtracking

Before our weekly session

Since this is a review and project week, we will not have any warm up questions that you will be required to submit. However, this is a great time for you to go back to topics and problems from the past few weeks that you had found more challenging.

It would also be advantageous to get an overview of what the project entails so you can ask questions early on. Aim to spend 5-10 hours on the project.

In class exercises

We will be working on more open ended programming problems in groups with a focus of understanding the spec and test case generation.

Create a new codepad document to work on.

For the following questions:

  1. Understand the problem. Besides the given example(s) try to come up with some of your own examples for the problem. Capture a useful universe of these examples as test cases.

  2. Match. Does this remind you of coding patterns you have used before, again most important to generate test cases from this.

  3. Prototype and implementation. Write code that addresses the test cases. Test driven development can be a very successful style for these problems.

You will be given 25 - 30 minutes per problem and work in groups of 2 - 3.

  1. Connect Four - Connect Four is a game where two players take turns dropping their color discs into a vertically suspended grid. The game ends when a player adds a disc to the playing grid that connects four discs of their color. The connected discs can be in a horizontal, vertical or diagonal line. Your task is to write a class that models the game. The main method of your class move needs to be able to 1) Print "Game Over" when there are no more moves possible, 2) Print "Player X" wins win either player 1 or player 2 "connect four".

  2. Simple Transaction Database - Create a simple key/value in memory database. The database must support the following api:

    1. GET key - returns the value stored in key
    2. SET key value - sets the value for key to be value
    3. COUNT value - gives the count for the number of values
    4. DELETE key - deletes the key
    5. BEGIN begins a transaction
    6. ROLLBACK rollsback the current transaction
    7. COMMIT - makes all current transactions (nesting is allowed) permanent.

    E.g.,

    SET a 10
GET a # => 10
COUNT 10 # => 1
BEGIN
SET a 20
BEGIN
COUNT 10 # => 0
DELETE a
COUNT 20 # => 0
ROLLBACK
COUNT 20 # => 1
ROLLBACK
GET a # => 10

  3. Filter Language - You will create a simple data structure that is a list of records. The records have simple string value fields. The fields are a, b, c. The values will only be floating point. You will create a function that will take the records list that you create and also a simple json description of filter operations. You need to design your json schema to support the following abilities:

    1. Support filtering with !=, ==, <, > on any of the fields
    2. support boolean logic AND and OR combining your filters
    3. not every record will have all fields.

    E.g.,

    Your json data structure should support the following type of filtering:

    a < 10 AND b != 5 OR c > 2

    And you need to demonstrate this on a dataset that you write as well. IO is not important, i.e., you can just instantiate these data sets in memory.

Mock Interview

This segment of the class will be a practice interviewing session where everyone will be paired up with a partner. This guide gives an insight on how to:

  1. Conduct a productive mock interview (when playing the role of the interviewer)
  2. Be a proactive interviewee

You should spend 30 minutes on one problem before switching roles with your partner and moving on to their question.

First, introduce yourself to your partner. Then, one person should start off as the interviewer, and share the collabedit link they created before the session, where the two of you will collaborate.

Challenge Questions:

  1. Longest Consecutive Sequence
  2. Maximal Rectangle

Role of the interviewer:

  • Help the interviewee understand the problem and answer any clarifying questions
  • Provide useful hints to help move the problem along
  • Help the interviewee debug their code once they’ve mapped out a solution

What the interviewer should NOT do:

  • Not pay attention while the interviewee is working through their solution
  • Blatantly tell the interviewee the most optimal solution within 10 minutes of them being stuck. Instead, try to provide helpful hints that will guide them in the right direction
  • Take over typing for the interviewee because they are stuck

Role of the interviewee:

  • Understand the question being asked and think through the trade-offs of different approaches
  • Work through the problem and come to a working solution

What the interviewee should NOT do:

  • Immediately start writing code after reading the question. Instead, ask clarifying questions and discuss the approach you have in mind. Talk out loud about possible approaches and their tradeoffs. Think through a couple test cases to see if your approach would work.
  • Stay silent for more than 3 minutes. If you find yourself having a mental block, try to articulate what part you’re stuck on. Perhaps there’s certain terminology you’re not confident with or you can’t visualize a part of the problem you’re trying to solve. Talking out loud helps your interviewer guide you.
